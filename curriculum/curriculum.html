UAV Student — Учебная программа

Инженерный курс по Betaflight, INAV, ArduPilot, PX4, интеграции companion computer, программируемой логике, работе с исходниками, автономии и эксплуатационной надёжности.

32–56 недель База → BF → INAV → ArduPilot → PX4

Идея программы курса (не часть сайта): папка curriculum/ — только HTML+CSS. Не входит в проект index.html + style.css. Тексты директив и ТЗ: единый файл PROJECT-DIRECTIVES.md в корне (отдельные *.md — указатели).

Цели и уровни подготовки

Уровень 1 — Оператор

  • Прошивка, порты, датчики, режимы, failsafe, логирование.
  • Базовый тюнинг и безопасный запуск.

Уровень 2 — Интегратор

  • GPS, компас, телеметрия, VTX, rangefinder, сервы.
  • Companion computer и внешние MCU (ESP32/STM32).

Уровень 3 — Автоматизатор

  • Lua (ArduPilot), IPF/JS (INAV), CLI-профили (BF).
  • Offboard/ROS 2 режимы и внешняя логика.

Уровень 4 — Разработчик

  • Сборка из исходников, feature selection, кастомизация.
  • Модули PX4, OEM-подход, анализ архитектуры flight stack.

Архитектура знаний (3 уровня)

Нелинейная модель: концепты → стек → проверяемая практика.

Уровень 1 — Фундамент

  • PID, rates, фильтры, базовая динамика полёта.
  • Сенсоры: gyro, accel, baro, compass, GPS.
  • Протоколы: UART, I2C, SPI, CRSF/ELRS, MAVLink.

Уровень 2 — Прошивки

  • ArduPilot: миссии, Lua, companion integration.
  • INAV: навигация, wing, IPF/JS-логика.
  • Betaflight: low-level control, CLI, Blackbox.
  • PX4: uORB, control allocation, ROS 2.

Уровень 3 — Практика

  • Настройка реального дрона под задачу.
  • Логи, диагностика и разбор симптомов.
  • Фейлы, дебаг, повторяемый результат.

Методология обучения

  • Изучение по слоям системы: железо, сенсоры, управление, навигация, логи, интеграция, код.
  • Каждая тема: концепция → внутреннее устройство → настройка → диагностика → мини-проект.
  • Правило безопасности: сначала simulation/SITL, потом реальные полёты.
  • Каждая неделя даёт артефакты: чеклист, заметка, конфиг/скрипт, черновик страницы сайта.

Режим миссий

Каждый учебный блок — миссия с измеримым входом и выходом.

Миссия Цель Вход Проверка/Выход
Настроить PID Стабильный полёт без выраженных осцилляций. Дрон + базовая прошивка + безопасный полигон. Видео полёта + diff/params + лог до/после.
Подключить GPS Устойчивый GPS fix и корректные координаты. FC + GPS + свободный UART. Координаты и спутники в конфигураторе/GCS.
Проверить failsafe Прогнозируемое поведение при потере связи. Настроенный профиль failsafe. Протокол теста и подтверждённый сценарий.

Система проверки знаний

Чек-лист модуля

  • Сделан backup/diff/params до изменений.
  • Выполнен минимум один практический полёт/симуляция.
  • Сохранён лог и сделаны выводы.

Контрольные вопросы

  • Что такое D-term и как влияет на поведение?
  • Почему появляются осцилляции?
  • Чем ArduPilot отличается от Betaflight по назначению?

Мини-тесты

  • 10–15 вопросов после каждого крупного этапа.
  • Порог прохождения: не ниже 80%.
  • Если ниже порога — повтор темы и миссии.

Режим лаборатории

Экспериментальная ветка: причинное понимание, а не «настроил по гайду».

  • Сравнить PID на 5" и 10" платформах.
  • Сравнить фильтры по логам вибраций и motor traces.
  • A/B тест двух прошивок на одном дроне.
  • Серия тестов: одно изменение за итерацию.

Причинно-следственные связи

Для каждого параметра — не только «что включить», но и логика влияния на полёт.

Параметр Что делает Когда применять Симптомы неправильной настройки
D-term Демпфирует быстрые изменения ошибки. При bounce back и резких остановках. Перегрев моторов, шум, дрожание.
Notch filter Подавляет узкополосные вибрации. При выраженных вибрационных пиках в логах. Осцилляции, «грязный» gyro сигнал.

Журнал ошибок и фейлов

Самый ценный раздел: реальные поломки и восстановление.

Что не работало Причина Как починил Как предотвращать
Нет GPS фикса Неверный UART/baudrate, помехи. Переназначен порт, перепроверен wiring. Шаблон проверки портов перед полётом.
Осцилляции после тюна Агрессивный D-term и слабая фильтрация. Откат параметров, анализ Blackbox, пошаговый тюн. Менять по одному параметру за итерацию.

Основные треки прошивок

Трек A — Betaflight (6–8 недель)

  • Базовая эксплуатация, Ports/Receiver/Modes/OSD.
  • PID, filters, feedforward, RPM filtering.
  • Blackbox анализ и симптомный тюнинг.
  • CLI: diff/dump/get/set/save, resource remapping.
  • Custom builds под ограничения FC.

Трек B — INAV (6–8 недель)

  • Навигационные режимы и миссии для квадро/крыла.
  • Fixed-wing setup: launch, cruise, RTH, geozones.
  • Programming Framework: logic conditions, variables.
  • JS-слой для пользовательской логики поведения.

Трек C — ArduPilot (8–10 недель)

  • Copter/Plane экосистема, миссии, EKF, логи.
  • Окто/X8, heavy-lift, отказоустойчивость.
  • Lua scripting в sandbox, applets, параметры.
  • MAVLink + companion computer (Raspberry Pi).
  • Build from source, OEM customization.

Трек D — PX4 (8–10 недель)

  • Модульная архитектура: modules, uORB, work queues.
  • Actuator setup и control allocation для квадро/окто.
  • ROS 2 и offboard/external modes.
  • Разработка собственного модуля PX4.

Workflow добавления статьи (актуально)

Источник правды: content/*.md + data/content-manifest.json. Сборка обновляет index.html только между маркерами AUTO:TOC:*, AUTO:ARTICLES:*, AUTO:PROGRESS:*, поэтому ручное копирование HTML не требуется.

1. Создать статью одной командой

python scripts/new_article.py inav inav-gps-fix "GPS Fix и диагностика" L2

Скрипт создаёт .md, добавляет запись в манифест и сразу запускает автопубликацию.

2. Заполнить содержание

Шаблон: content/article-template.md. Обязательные секции для публикации: summary, theory, practice, diagnostics, references.

3. Проверить и пересобрать безопасно

publish-safe.bat

Порядок: проверка Python → валидация → публикация в index.html с понятными ошибками.

4. Режимы черновика

python scripts/new_article.py inav inav-test "Тестовая тема" L1 --status draft --no-publish

Используйте draft для незавершённых материалов; такие статьи не попадают в обычную публикацию.

Минимальный рабочий сценарий

  • Создать статью через new_article.py.
  • Заполнить текст и источники в .md.
  • Запустить publish-safe.bat и проверить результат в index.html.
  • После проверки — коммит/push через publish-git.bat или publish-git.sh.

Скрипты проекта: что делает каждый и как запускать

Ниже — практическая шпаргалка по всем основным скриптам. Команды запускать из корня проекта UAV_Student.

scripts/new_article.py

Для чего: создать новую статью, добавить её в манифест, при необходимости сразу опубликовать.

python scripts/new_article.py px4 px4-vision "Компьютерное зрение в PX4" L3

Черновик без публикации:

python scripts/new_article.py px4 px4-vision "Компьютерное зрение в PX4" L3 --status draft --no-publish

scripts/validate_content.py

Для чего: проверить манифест, уникальность id, наличие файлов и обязательных секций.

python scripts/validate_content.py

Если есть ошибки, публикация должна быть остановлена до исправления.

scripts/publish.py

Для чего: собрать TOC, прогресс и HTML-статьи в index.html.

python scripts/publish.py

Полезные режимы:

python scripts/publish.py --include-drafts
python scripts/publish.py --dry-run

publish-safe.bat

Для чего: безопасный сценарий «Python → валидация → публикация» с понятными сообщениями об ошибках.

publish-safe.bat

publish.bat / publish.sh

Для чего: быстрый ручной запуск публикации без дополнительной диагностики.

publish.bat
bash publish.sh

publish-git.bat / publish-git.sh

Для чего: собрать сайт и выполнить git-цикл (commit/push) в одном запуске.

publish-git.bat
bash publish-git.sh

scripts/generate_content_stubs.py

Для чего: массово создать недостающие content/*.md по манифесту.

python scripts/generate_content_stubs.py

scripts/build_content.py

Для чего: служебная сборка TOC-файлов в build/ и техобновление манифеста из _manifest_data.py.

python scripts/build_content.py
python scripts/build_content.py --write-manifest

Чёткий алгоритм: как добавить статью

  1. Создайте статью: 
    python scripts/new_article.py inav inav-gps-fix "GPS Fix и диагностика" L2
  2. Откройте созданный файл content/inav/inav-gps-fix.md и заполните разделы: summary, theory, practice, diagnostics, references.
  3. Проверьте качество данных: 
    python scripts/validate_content.py
  4. Опубликуйте безопасно: 
    publish-safe.bat
  5. Проверьте результат в браузере: статья должна появиться в нужной вкладке и в её TOC.
  6. Если статья не появилась, проверьте в первую очередь: статус статьи в data/content-manifest.json (должен быть published) и отсутствие ошибок валидации.

Алгоритм для черновика (без публикации)

  1. Создать черновик: python scripts/new_article.py inav inav-test "Тестовая статья" L1 --status draft --no-publish
  2. Доработать текст в content/*.md.
  3. Когда готово — сменить статус на published в манифесте.
  4. Запустить publish-safe.bat.

Единый стандарт курса и обучающей страницы

Стандарт для новых материалов: курс остаётся цельным и сравнимым между прошивками.

1) Структура курса

  • Уровень 1: фундамент (динамика, сенсоры, протоколы).
  • Уровень 2: прошивки (Betaflight, INAV, ArduPilot, PX4).
  • Уровень 3: практика (миссии, логи, дебаг, фейлы, лаборатория).
  • Каждый модуль завершается измеримым результатом: лог, видео, конфиг, выводы.

2) Формат учебной страницы

article.card[data-level][data-added]

h2 + краткое вводное p.muted

.article-meta (уровень, время, версия)

h3: Теория → Практика → Диагностика

.cli-block (если есть команды)

.references + .related-topics + .checklist

.article-footer (добавлено/обновлено/уровень)

3) Семантика HTML

  • Одна страница = main с набором section.
  • Тема = article, группы карточек = div.grid.
  • Сравнения через table с thead/tbody.
  • Иерархия: один h1, далее h2 → h3 без пропусков.

4) CSS

  • Токены в :root; компоненты: .card, .grid, .tag.
  • Модификаторы: .ok, .warn; без inline-стилей.

5) Контент

  • Тон инженерный, воспроизводимые инструкции, версии прошивок в модулях.
  • Фейлы: что сломалось / почему / как исправлено / профилактика.
  • Обязателен чек-лист публикации: структура, ссылки, техника, интеграция в глоссарий/changelog.

6) Адаптивность

  • Брейкпоинты: до 480px, до 900px, выше 900px.
  • Узкие экраны: одна колонка, таблицы с горизонтальным скроллом.

7) Жизненный цикл статьи

Идея → new_article.py → заполнение content/<stack>/<id>.md → чек-лист и источники (data/references.json или archive/references.json, опционально) → publish-safe.bat → проверка статьи и TOC в index.html → ревью → запись в раздел Changelog файла PROJECT-DIRECTIVES.md.

8) Единый документ директив

Полные тексты концепции, методологии, гайдов и истории изменений — в PROJECT-DIRECTIVES.md. Корневые project-concept.md, content-methodology.md и др. — короткие указатели на разделы этого файла.

Сырые тексты ТЗ — archive/tz-texts/ (см. archive/README.md). Папка curriculum/ — отдельно от сайта.

Источник правды по текстам ТЗ и директив: PROJECT-DIRECTIVES.md (единый файл с оглавлением). Корневые README.md, project-concept.md и остальные *.md — краткие указатели на разделы единого документа. Эта вкладка даёт обзор и состояние проекта; при расхождении обновлять сначала PROJECT-DIRECTIVES.md, затем при необходимости сводку здесь.

PROJECT-DIRECTIVES.md — единый документ

5) Что считается «ядром» проекта

  • Рендер сайта: index.html + style.css.
  • Контент: content/ + data/content-manifest.json.
  • Публикация: scripts/validate_content.py + scripts/publish.py (или publish-safe.bat).
  • Директивы: PROJECT-DIRECTIVES.md.

Концепция проекта (в PROJECT-DIRECTIVES.md)

Назначение: образовательный веб-ресурс по разработке, настройке и эксплуатации БПЛА. Полный текст — в PROJECT-DIRECTIVES.md, раздел project-concept. Файл project-concept.md — указатель.

Формат: статическая площадка с табами и оглавлением/якорями; источники по политике источников (в едином документе).

Принцип: JS не обязателен, навигация должна деградировать корректно (content-first).

Методология контента (в PROJECT-DIRECTIVES.md)

Полный процесс добавления статей: цель, принципы, иерархия, уровни L1–L4, жизненный цикл, шаблон HTML, чек-лист публикации, UI-паттерны. Файл content-methodology.md в корне — указатель на этот раздел.

Принципы: шаблонность, иерархичность, верифицируемость, прогрессивность, связность.

Жизненный цикл: идея → место в структуре → шаблон → чек-лист → archive/references.json (опционально) → вёрстка → глоссарий/связи → ревью → Changelog в PROJECT-DIRECTIVES.md.

HTML-шаблон: .article-meta, .references, .related-topics, .checklist, .article-footer; опционально .breadcrumb, .section-progress, .orientation-guide.

Статус: curriculum/ — прототип идеи курса, не часть сборки index.html.

content-guidelines.md — правила контента

ПринципОписание
Слоистый подходОт фундамента (сенсоры, PID) к прикладным слоям и кастомизации
Причинно-следственное обучениеПараметр: что делает → когда применять → симптомы ошибки
Практическая верификацияКаждый модуль завершается измеримым результатом
БезопасностьСначала симуляция/SITL, затем реальные полёты

Каркас модуля: теория → практика → диагностика → ссылки → чек-лист. Уровни L1–L4 и критерии — в документе.

article-template.md — шаблон статьи

Шаблон для черновиков .md (в корне или в content/): кратко, что это, зачем, как работает, практика, типичные ошибки, диагностика, источники, смежные темы.

Для HTML-вставки в сайт используйте разметку из content-methodology.md (§3.2) с обязательными блоками и классами.

source-policy.md — политика источников

A (официальные): документация, репозитории, wiki, maintainer docs, release notes, спецификации.

B (вторичные авторитетные): статьи разработчиков, technical blogs, проверенные разборы.

C (community): форумы, Reddit, Discord/Discourse, issues, личные гайды — осторожно.

Правило приоритета: A важнее всех; B/C только как дополнение. Для C обязательно указывать версию и необходимость сверки с A.

references-db.md — база ссылок

Единый формат записей для references.json: id, title, url, stack, type, trust, uiTag, notes, lastChecked, firmwareNote.

UI-теги primary/community/video не заменяют уровни доверия A/B/C.

{
  "id": "bf-docs-main",
  "title": "Betaflight — документация",
  "url": "https://betaflight.com/docs/development",
  "stack": "betaflight",
  "type": "official_docs",
  "trust": "A"
}

visual-style.md — визуальный стиль

Идея: dark technical reference interface — спокойный инженерный интерфейс, где оформление не спорит с контентом.

Блоки: hero, toc, card, callout-info, callout-warn, doc-links, community-links, compare-table, checklist, glossary-entry.

Таблицы: thead/tbody; на мобильных — скролл или карточки строк. Контрольные ширины: 480px, 900px, выше 900px.

style-guide.md — код и UI

Классы по назначению, данные в references.json по references-db.md. Токены и компоненты для index.html + style.css.

Компоненты: TabBar, Card, Grid, Table (с data-label), Tag, DocLink, Accordion. JS — опционально при CSS-fallback.

История изменений (в PROJECT-DIRECTIVES.md)

Актуальный changelog ведётся в разделе Changelog файла PROJECT-DIRECTIVES.md. В том числе: наполнение вкладок index.html, глоссарий и журнал ошибок, ресурсы и чек-листы, объединение markdown-документации в единый файл, обновление curriculum.html.

Полный changelog ведётся в PROJECT-DIRECTIVES.md; это актуальный источник истории изменений.

roadmap-notes.md — заметки по дорожной карте

Рабочее место для идей по этапам, срокам и зависимостям. Не дублирует таблицу из curriculum.html; хранит риски и открытые вопросы.

Правила: кратко, с ссылками по source-policy.md; после стабилизации переносить выводы в контент/документы и удалять шум.

structure-drafts.md — черновики структуры курса

Эксперименты с иерархией разделов и навигацией. Это не финальная карта сайта.

Перед переносом в контент согласовывать с content-methodology.md.

Конец обзора. Источник правды по текстам: PROJECT-DIRECTIVES.md.
