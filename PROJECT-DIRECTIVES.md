# UAV Student — концепция проекта

Единый вектор развития: согласован с **`architecture.md`**, **`content-methodology.md`**, **`source-policy.md`**. Структура репозитория **временно плоская** (корень проекта) — см. **`architecture.md`**.

## 1. Общее описание

### 1.1 Назначение

**UAV Student** — образовательный веб-ресурс для инженерной подготовки в области разработки, настройки и эксплуатации БПЛА. Охват стека: от базовой настройки полётных контроллеров до кастомизации прошивок и интеграции с бортовыми компьютерами.

### 1.2 Целевая аудитория

- начинающие пилоты и операторы БПЛА;
- инженеры-интеграторы под конкретные задачи;
- разработчики автономных систем;
- студенты профильных специальностей;
- исследователи (ROS 2, ML на дронах).

### 1.3 Формат продукта

**Статическая** веб-площадка с глубокой навигацией: **табы** для крупных разделов и **якоря / оглавление** внутри раздела. Контент — достоверный, проверяемый, со ссылками на официальную документацию и вторичные источники по **`source-policy.md`**.

| **Проект сайта** | **`index.html`** + **`style.css`** (корень) | Вкладки по **стекам** (в перспективе): Betaflight, INAV, ArduPilot, PX4, Companion. См. раздел 6. |

Обе схемы **последовательны**: папка **`curriculum/`** — идея курса; **`index.html`** + **`style.css`** — сайт. Подробнее — **`architecture.md`**.

## 3. Целевая файловая структура сайта

Сайт собирается из **двух файлов** в корне:

```
index.html
style.css
```

Опционально: **`archive/references.json`** (формат — раздел *references-db* ниже). Сырые тексты ТЗ — **`archive/tz-texts/`** (временно).

**Принцип:** разметка и стили без обязательного JS. При необходимости позже — скрипты и **graceful degradation** (например, паттерн CSS+radio как в `curriculum/curriculum.html`). См. `content-methodology.md` (content-first).

Черновики статей — отдельные `.md` или HTML; перенос в `index.html` — по мере готовности (запись в раздел Changelog **`PROJECT-DIRECTIVES.md`**).

## 4. Методология обучения

Сводно изложена в **`content-guidelines.md`** (принципы, шаблон модуля, уровни L1–L4). Здесь — только тезисы:

- **Слоистый подход:** фундамент → прикладные слои → кастомизация.
- **Причинно-следственность:** параметр → эффект → симптомы ошибки.
- **Верификация:** чек-лист, лог, diff, видео по модулю.
- **Безопасность:** SITL/симуляция до реального полёта.

## 6. Структура контента продукта

| Таб | Содержание | Ориентир объёма |
|-----|------------|-----------------|
| Betaflight | FPV, PID, фильтры, Blackbox, CLI | 6–8 модулей |

| ArduPilot | Миссии, Lua, EKF, тяжёлые платформы, companion | 8–10 модулей |

Сравнение прошивок, журнал ошибок, глоссарий, инструментарий — внизу или отдельными секциями; см. **`style-guide.md`** (компоненты).

Блок «Источники» в каждом модуле; теги `primary` / `community` / `video` не заменяют уровни A/B/C из **`source-policy.md`** — их стоит **сопоставлять** (например, primary ≈ A при официальном URL).

## Раздел: content-methodology — методология контента

# Методология добавления контента — UAV Student

Документ задаёт **устойчивый, повторяемый процесс** добавления статей и модулей. Детальные правила текстов и уровней — **`content-guidelines.md`**. Источники — **`source-policy.md`**. Каркас темы в Markdown — **`article-template.md`**. Концепция сайта — **`project-concept.md`**.

Статус папки **`curriculum/`** (идея курса, не сборка `index.html`) — см. раздел **«Приложение: curriculum/»** внизу.

---

## 1. Цель и принципы

### 1.1 Цель

Создать процесс добавления новых статей и модулей, который обеспечивает:

- единообразие структуры и стиля;
- предсказуемость навигации для пользователя;
- соблюдение стандартов из `content-guidelines.md` и `source-policy.md`;
- привычный паттерн взаимодействия с интерфейсом у студента.

### 1.2 Ключевые принципы

| Принцип | Описание |
|----------|----------|
| **Шаблонность** | Каждый новый материал создаётся по единому шаблону |
| **Иерархичность** | Чёткое место каждого материала в структуре проекта |
| **Верифицируемость** | Каждый материал проходит чек-лист перед публикацией |
| **Прогрессивность** | Материалы добавляются по уровням сложности (L1 → L4) |
| **Связность** | Новые статьи имеют перекрёстные ссылки на существующие |

---

## 2. Иерархия контента

### 2.1 Структура проекта (контентная)

```text
UAV Student
│
├── Разделы (табы)
│   ├── Betaflight
│   │   ├── Модули (группы статей)
│   │   │   ├── Введение в Betaflight (L1)
│   │   │   ├── Настройка портов и приёмника (L1)
│   │   │   ├── Режимы и OSD (L1)
│   │   │   ├── PID-тюнинг (L2)
│   │   │   ├── Фильтры и Blackbox (L2)
│   │   │   ├── CLI и ремаппинг (L3)
│   │   │   └── Кастомная сборка (L4)
│   │   └── Каждая статья = один HTML-блок .card
│   │
│   ├── INAV (аналогично)
│   ├── ArduPilot
│   ├── PX4
│   └── Companion
│
├── Общие блоки (сквозные)
│   ├── Глоссарий
│   ├── Журнал ошибок
│   └── Сравнение прошивок
│
└── Вспомогательные
    ├── Инструментарий (таб)
    └── Чек-листы (общие)
```

### 2.2 Уровни статей (L1–L4)

| Уровень | Характер контента | Место в структуре |
|---------|-------------------|-------------------|
| **L1 — Оператор** | Базовая настройка, первые полёты | Начало раздела, первые модули |
| **L2 — Интегратор** | GPS, телеметрия, внешние устройства | Середина раздела |
| **L3 — Автоматизатор** | Скриптинг, миссии, offboard | Продвинутая часть |
| **L4 — Разработчик** | Исходный код, кастомизация | Заключительная часть |

### 2.3 Именование `id` статей и якорей (дисциплина)

Чтобы не получить «музей» из случайных якорей и дублей, придерживайтесь единой схемы. **Латиница, нижний регистр, дефисы**; без кириллицы, дат, версий прошивки в `id`, без `article1`, `new`, `final-final`.

**Основные статьи (одна карточка = один `article`):**

```text
<stack>-<topic>
```

**Стеки (`<stack>`):** `bf`, `inav`, `ap`, `px4`, `comp`, `tools` (коротко, как в URL-привычке).

**Примеры:** `bf-intro`, `bf-pid`, `bf-blackbox`, `inav-gps`, `inav-wing`, `ap-lua`, `px4-architecture`, `px4-control`, `comp-esp32`.

**Вспомогательные блоки secondary / заметки внутри статьи** (не отдельные карточки на уровне таба):

```text
<stack>-<topic>-note-<slug>
```

**Примеры:** `inav-gps-note-oscar-liang-basics`, `bf-pid-note-bardwell-filtering`, `bf-blackbox-note-oscar-liang-blog`.

**Чек-листы и лабораторные внутри темы** (если выносите в отдельный якорь/блок):

```text
<stack>-<topic>-checklist
<stack>-<topic>-lab
```

**Примеры:** `inav-gps-checklist`, `px4-architecture-lab`.

**Не использовать в `id`:** даты, номера версий в имени, длинные названия статьи, кириллицу, общие слова без привязки к теме.

**Оглавление раздела (`nav.toc`):** рекомендуемый формат ссылки — **номер · уровень L1–L4 · короткая подпись** (классы `.toc__num`, `.toc__level`, `.toc__label`, на контейнере — `.toc--enhanced`). Образец: раздел Betaflight в `index.html`. Остальные вкладки можно перевести на тот же формат постепенно.

### 6.1 Предлагаемая структура

Имена файлов черновиков **совпадают с `id` статьи** в `index.html` (см. §2.3), чтобы не плодить расхождений при сборке.

```text

├── index.html
├── style.css
├── content/                      # черновики → потом сборка в index.html
│   ├── betaflight/
│   │   ├── bf-pid.md
│   │   ├── bf-blackbox.md
│   │   └── bf-custom.md
│   ├── inav/
│   │   ├── inav-gps.md
│   │   ├── inav-wing.md
│   │   └── inav-missions.md
│   ├── ardupilot/
│   ├── px4/
│   ├── companion/
│   ├── tools/
│   │   └── tools-overview.md
│   └── notes/                    # опционально: обзоры secondary, не дублируя чужие тексты
│       ├── oscar-liang/
│       └── joshua-bardwell/
├── data/
│   ├── content-manifest.json     # источник правды: таб, id, порядок TOC, путь к .md
│   └── references.json           # опционально: общий реестр ссылок (может дублировать archive/)
├── scripts/
│   ├── build-content.py          # пока: TOC по manifest; далее — рендер в HTML
│   ├── validate-content.py       # id, файлы published, related/next, обязательные секции .md
│   ├── generate_content_stubs.py # черновики .md для записей без файла
│   └── _manifest_data.py         # генерация manifest (опционально)
├── archive/references.json       # исторический реестр (см. раздел *references-db*)
└── PROJECT-DIRECTIVES.md
```

Папка **`content/`** может быть создана заранее (пустые каталоги + `.gitkeep`) или по мере появления материалов. **Уже сейчас** держите именование так, как будто структура существует — иначе позже придётся выковыривать десятки фрагментов из одного `index.html`.

### 6.2 Процесс через Markdown

1. Создать `.md` в `content/<раздел>/` по шаблону **`content/article-template.md`** или командой **`python scripts/new_article.py`**; имя файла = **`id`** карточки (`bf-pid.md` и т.д.).
2. Добавить статью в **`data/content-manifest.json`** (таб, `id`, `order`, `sourceFile`, статус).
3. Запустить **`python scripts/validate_content.py`** или сразу **`python scripts/publish.py`** (валидация + вставка в `index.html` по маркерам `AUTO:TOC:*` / `AUTO:ARTICLES:*` / `AUTO:PROGRESS:*`).
4. Добавить записи в **`data/references.json`** или **`archive/references.json`** (если ведёте реестр).
5. Собрать HTML в **`index.html`** (пока вручную; далее — расширение `build-content.py`).

### 6.3 Контент-пайплайн (этап 11)

**Источник правды по навигации:** `data/content-manifest.json` (описание вкладок и статей). **Источник правды по тексту модуля:** `content/<stack>/<id>.md` с YAML front matter и секциями `summary` / `theory` / `practice` / `diagnostics` / `references` (плюс опционально `checklist`, `lab`, `selfcheck`, `notes`). Скрипт **`validate-content.py`** проверяет уникальность `id`, наличие файла для `status: published`, небитые `related` / `next` и обязательные заголовки в теле.

### 6.4 Secondary-заметки в HTML

Блок **«Дополнительные заметки и разборы»** внутри статьи: секция **`.article-notes`**, внутри — **`<details class="source-accordion secondary-source">`** с **`id`** по схеме **`<stack>-<topic>-note-<slug>`**. Размещение: после **references**, перед **checklist**. Полный текст сторонних материалов не копировать; только тезисы + ссылка на оригинал + тег **secondary | B** (см. раздел *source-policy*).

Пример (живые образцы — `inav-gps`, `bf-blackbox` в `index.html`):

```html
<section class="article-notes" aria-labelledby="TOPIC-notes-title">
  <h4 id="TOPIC-notes-title">🗂 Дополнительные заметки и разборы</h4>
  <details id="inav-gps-note-oscar-liang-basics" class="source-accordion secondary-source">
    <summary>
      Oscar Liang · GPS setup basics
      <span class="tag">secondary | B</span>
    </summary>
    <div class="source-accordion__body">
      <p class="muted">Краткий разбор без копирования оригинала.</p>
      <ul class="note-points">…</ul>
      <div class="note-actions">
        <a href="URL" class="doc-link" target="_blank" rel="noopener noreferrer">Оригинал</a>
      </div>
    </div>
  </details>
</section>
```

## 2. Методология обучения

### 2.1 Базовые принципы

| Принцип | Описание |
|---------|----------|
| **Слоистый подход** | От фундамента (сенсоры, PID) к прикладным слоям (навигация, миссии) и к кастомизации (код, модули). |
| **Причинно-следственное обучение** | Каждый параметр: что делает → когда применять → симптомы при ошибке. |
| **Практическая верификация** | Модуль завершается измеримым результатом: чек-лист, лог, diff, видео. |
| **Безопасность прежде всего** | Сначала симуляция/SITL, затем полёт в контролируемых условиях. |

### 2.2 Структура учебного модуля

Каждый раздел (например, «Настройка PID в Betaflight») желательно строить по единому каркасу:

```markdown
# Название модуля

## Теория
- Что это и зачем
- Принцип работы / место в архитектуре

## Практика
- Оборудование и ПО
- Пошаговая инструкция
- CLI / параметры

## Диагностика
- Как проверить настройку
- Логи, Blackbox
- Типичные симптомы и причины

## Ссылки
- Официальная документация
- Видео и разборы (с пометкой уровня B/C)

## Чек-лист
- [ ] Backup конфигурации
- [ ] Настройка выполнена
- [ ] Тест полёта/симуляции
- [ ] Логи сохранены и проанализированы
- [ ] Выводы зафиксированы
```

Детальная версия с блоками «Смежные темы» и «Источники» — в **`article-template.md`**.

### 2.3 Уровни освоения

| Уровень | Навыки | Формат проверки |
|---------|--------|-----------------|
| **L1 — Оператор** | Прошивка, порты, режимы, failsafe, базовый тюнинг | Чек-лист + видео полёта |
| **L2 — Интегратор** | GPS, телеметрия, companion, внешние сенсоры | Конфигурационный проект |
| **L3 — Автоматизатор** | Lua/JS, миссии, offboard | Рабочий скрипт + демонстрация |
| **L4 — Разработчик** | Сборка из исходников, модули, ROS 2 | Репозиторий с изменениями |

## Раздел: source-policy — политика источников

# Source policy — политика источников

Все фактические технические утверждения должны опираться на **иерархию доверия**. Смешивать официальную документацию и случайный совет без пометки — нельзя.

---

## Уровень A — официальные источники

Используются как **основной эталон**:

- официальная документация прошивки / стека;
- GitHub-репозитории проекта;
- официальные wiki;
- maintainer docs;
- release notes;
- спецификации протоколов.

---

## Уровень B — авторитетные вторичные источники

Для пояснений, примеров, разборов:

- статьи опытных разработчиков;
- качественные technical blogs;
- проверенные видео-разборы;
- community knowledge base с устойчивой репутацией.

Помечать как **вторичные**, но допустимые при отсутствии противоречий с уровнем A.

---

## Уровень C — практические community-материалы

Использовать **осторожно**:

- форумы;
- Reddit;
- треды Discord / Discourse;
- GitHub issues;
- личные гайды.

Обязательно указывать:

- что это **вторичный / community** источник;
- что информация может зависеть от **версии** прошивки и железа;
- что нужно **сверять** с официальной документацией (уровень A).

---

## Правило приоритета

**Официальные источники (A) важнее всех остальных.**  
Неофициальные включаются как вспомогательные и **не заменяют** документацию A.

---

## Как оформлять спорные данные

- Явно писать «по состоянию на версию X», если важно.
- Если данные только из C — не выдавать их за общепринятый факт; при возможности подтвердить по A или B.

---

## Как ссылаться

- На документацию — **стабильные URL** разделов, где возможно.
- На GitHub — issue/PR с контекстом, зачем ссылка.
- Внутри статьи — блок **«Источники»** в конце (см. `article-template.md`).
- Реестр ссылок и JSON — **`references-db.md`** (теги UI `primary` / `community` не заменяют уровни A/B/C).

---

## Примечание для читателя (можно вставлять в статьи)

> Неофициальные материалы используются как вспомогательные и не заменяют официальную документацию.

## Раздел: references-db — база ссылок

# References database — база ссылок

Единый формат учёта источников для модулей курса и для файла **`archive/references.json`** (опционально). Уровни доверия **A / B / C** — в разделе *source-policy* ниже. UI-теги `primary` / `community` / `video` не заменяют A/B/C: их сопоставляют вручную при публикации.

---

## Поля записи

| Поле | Описание |
|------|----------|
| `id` | Стабильный идентификатор (slug) |
| `title` | Заголовок ссылки |
| `url` | HTTPS, предпочтительно постоянный раздел документации |
| `stack` | `betaflight` \| `inav` \| `ardupilot` \| `px4` \| `companion` \| `general` |
| `type` | `official_docs` \| `repo` \| `wiki` \| `video` \| `article` \| `forum` \| `issue` |
| `trust` | `A` \| `B` \| `C` |
| `uiTag` | опционально: `primary` \| `community` \| `video` — для отображения |
| `notes` | Кратко, зачем ссылка в контексте курса |
| `lastChecked` | Дата проверки актуальности (ISO: `YYYY-MM-DD`) |
| `firmwareNote` | Версии прошивки/инструмента, если критично |

---

## Пример JSON (`references.json`)

```json
[
  {
    "id": "bf-docs-main",
    "title": "Betaflight — документация",
    "url": "https://betaflight.com/docs/development",
    "stack": "betaflight",
    "type": "official_docs",
    "trust": "A",
    "uiTag": "primary",
    "notes": "Эталон по конфигуратору и разработке",
    "lastChecked": "2026-03-20",
    "firmwareNote": "Актуализировать под текущий мажор"
  }
]
```

---

## Правила

1. Любая новая фактическая ссылка в модуле должна появиться здесь или в JSON до/вместе с публикацией текста.  
2. Записи уровня **C** обязаны содержать `notes` о необходимости сверки с A.  
3. Дубли URL объединять по `id`, версии уточнять в `firmwareNote`.

## 5. Навигация (макеты)

**Десктоп (≥ 1024px):** шапка → ряд табов по стекам → колонка с оглавлением раздела + основной контент (теория, практика, диагностика, ссылки).

**Мобильный (≤ 768px):** табы — горизонтальный скролл или select; оглавление — аккордеон или FAB; таблицы — как в прототипе (`data-label`).

- Необязателен для базового чтения и для табов при использовании CSS-fallback (**`project-concept.md`**).

## Раздел: visual-style — визуальный стиль

# Visual style — стиль оформления

Документ фиксирует **дизайн-систему** для **`index.html`** + **`style.css`** и согласуется с направлением **`curriculum/curriculum.css`** (идея курса — отдельная папка).

**Токены, компоненты и HTML-примеры** — в **`style-guide.md`**. Общая концепция UI и навигации — **`project-concept.md`**.

## Общая идея

**Dark technical reference interface** — тёмная спокойная инженерная база знаний, а не постер «про дроны».

Правило: **оформление не должно спорить с контентом**.

---

## Характер

- тёмный фон;
- приглушённые акцентные цвета;
- высокий контраст основного текста;
- карточки для смысловых блоков;
- таблицы для сравнений;
- callout-блоки для предупреждений и важных замечаний;
- предсказуемые отступы и размеры (токены в `:root`).

---

## Цветовая логика (ориентир)

| Роль | Назначение |
|------|------------|
| Основной текст | Светлый нейтральный |
| Вторичный текст | `muted` |
| Ссылки | Мягкий синий / accent |
| Успех, корректный результат | Зелёный |
| Предупреждение, риск | Тёплый жёлтый |
| Критическая ошибка | Красноватый акцент — **редко**, только для реального риска |

Точные значения лучше закрепить в одном месте (CSS variables в `style.css`).

---

## Типы UI-блоков (единый набор)

Имена — **по назначению**, не «синяя плашка».

| Класс / блок | Назначение |
|--------------|------------|
| `hero` | Вводный блок раздела |
| `toc` | Навигация по разделу (оглавление) |
| `card` | Обычный контентный блок |
| `callout-info` | Важное пояснение |
| `callout-warn` | Риск, ошибка, ограничение |
| `doc-links` | Официальные источники |
| `community-links` | Вторичные источники (с пометкой) |
| `compare-table` | Таблица сравнения |
| `checklist` | Практические шаги / критерии |
| `glossary-entry` | Карточка термина |

Реализация в HTML/CSS может отличаться, но **семантика** должна совпадать.

---

## Типографика и отступы

- Базовый кегль и межстрочный интервал — читаемость на длинных текстах.
- Иерархия заголовков без пропусков уровней (`h1` → `h2` → `h3`).
- Единая сетка отступов между секциями и внутри карточек.

---

## Таблицы

- Заголовок через `thead`, тело через `tbody`.
- На узких экранах — горизонтальный скролл или раскладка «карточка строки» с подписями полей (как в прототипе).

---

## Адаптивность

Контрольные ширины (ориентир из прототипа): до **480px**, до **900px**, выше **900px**.  
Новые блоки не добавлять без проверки на типичных ширинах экрана.

---

## Связь с контентом

Визуальные блоки **doc-links** и **community-links** обязаны дублировать логику **`source-policy.md`**: официальное отдельно от вторичного.
